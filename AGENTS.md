# AGENTS.md — контекст для Codex / Claude Code

Это инструкция для ИИ-ассистента (Codex), который работает с этим проектом. Прочитай её целиком
перед первой задачей. Здесь — что это за проект, какие правила и какими командами всё делается.

## Что это за проект

Учебный продукт ученика лагеря **nFactorial Teens**. Один разработчик (подросток 14–16 лет) строит
свой продукт с твоей помощью. Объясняй кратко и по-доброму, не вали жаргоном.

**Стек (не меняй его):**
- **Vite + React 18 + TypeScript** — фронтенд (папка `src/`)
- **Supabase** — база данных, авторизация, хранилище файлов (клиент в `src/lib/supabase.ts`)
- **Vercel** — хостинг, авто-деплой при `git push`
- **git + GitHub** — версии и облако

## Железные правила

1. **Только Supabase для данных.** Клиент уже создан в `src/lib/supabase.ts`. Используй его. Не
   добавляй другую базу, другой backend, другой ORM.
2. **Схема базы — только через миграции, НЕ вручную.** Любое изменение базы (новая таблица,
   колонка, политика) = новый файл в `supabase/migrations/`, затем `npm run db:push`. Никогда не
   проси пользователя «вставить SQL в Supabase SQL Editor вручную».
3. **RLS обязателен.** На каждую новую таблицу включай Row Level Security и политики
   «пользователь видит только свои строки» (`auth.uid() = user_id`). Образец — в
   `supabase/migrations/20260601000000_init.sql`.
4. **Секреты — только в окружении.** Ключи (`VITE_SUPABASE_*`) живут в `.env.local` (git его
   игнорит) и в настройках Vercel. Никогда не вписывай ключи в код и не коммить их. Ключи
   сторонних AI-моделей (OpenAI/DeepSeek) держи в Supabase Edge Function, не на фронте.
5. **TypeScript, без `any`.** Маленькие компоненты в `src/components/`. Перед завершением задачи
   убедись, что `npm run build` проходит.
6. **Одна задача — одно изменение.** Не делай 10 несвязанных правок за раз. После изменения
   кратко скажи, что поменял и в каких файлах.
7. **Чистый и структурированный код (ВАЖНО).** Не сваливай всё в один огромный файл. Держи
   проект аккуратным — это правила ниже.

## Чистый код — структура (соблюдай всегда)

Цель: чтобы подросток мог открыть проект через месяц и легко разобраться. Никаких файлов-«простыней».

- **Маленькие файлы.** Один файл = одна понятная вещь. Держи файл **до ~150 строк**. Если файл
  растёт больше — раздели его на части. **Никогда не пиши тысячи строк в одном файле.**
- **Один компонент = один файл** в `src/components/`. Большой экран разбивай на маленькие
  компоненты (например `QuizScreen` → `QuestionCard`, `ScoreBar`, `ResultScreen`).
- **Выноси повторяющееся.** Если копируешь один и тот же код 2+ раза — сделай из него отдельный
  компонент или функцию.
- **Логику базы — в отдельные функции.** Запросы к Supabase складывай в `src/lib/` (например
  `src/lib/quiz.ts`), а не разбрасывай по компонентам.
- **Понятные имена.** `QuestionCard`, `loadQuestions`, `isCorrect` — сразу ясно, что это. Никаких
  `data2`, `tmp`, `comp1`.
- **Папки по смыслу.** UI → `src/components/`, работа с данными/Supabase → `src/lib/`,
  целые экраны → `src/pages/` (создай папку, если нужно).
- **Без мусора.** Удаляй неиспользуемый код, лишние `console.log`, закомментированные куски.
- После большой фичи спроси себя: «есть ли файл длиннее ~150 строк или повтор кода?» — если да,
  разбей, прежде чем считать задачу готовой.

## Все команды проекта

| Команда | Что делает |
|---------|------------|
| `npm install` | поставить зависимости (один раз) |
| `npm run dev` | запустить приложение локально (обычно http://localhost:5173) |
| `npm run build` | собрать прод-версию + проверить типы (запускай перед коммитом) |
| `npm run preview` | посмотреть собранную прод-версию локально |
| `npm run db:login` | Codex запускает; ребёнок только подтверждает вход в браузере |
| `npm run db:link -- --project-ref REF` | Codex берёт REF из URL; ребёнок вводит пароль БД, только если терминал спросит |
| `npm run db:new -- имя` | создать новый файл миграции в `supabase/migrations/` |
| `npm run db:push -- --dry-run` | Codex предварительно проверяет список миграций |
| `npm run db:push -- --yes` | Codex применяет проверенные миграции к облачной базе |
| `git add -A && git commit -m "..."` | сохранить версию (сейв) |
| `git push` | отправить в GitHub → Vercel сам пересоберёт сайт |

## Доступ Codex к Supabase CLI

Codex выполняет **все команды сам**. Ребёнок не должен копировать команды, искать project ref,
создавать access token или дебажить миграции.

1. Codex запускает `npm run db:login`. Ребёнок только входит в **свой** Supabase в открывшемся
   браузере и подтверждает доступ.
2. Codex сам извлекает `project ref` из `VITE_SUPABASE_URL=https://REF.supabase.co` и запускает
   `npm run db:link -- --project-ref REF`.
3. Если терминал запросил пароль БД, попроси ребёнка ввести его **прямо в терминал**. Не проси
   отправлять пароль в чат и не записывай его в файл.
4. Сначала `npm run db:push -- --dry-run`. Проверь, что применяются только ожидаемые миграции,
   новые таблицы включают RLS и политики доступа корректны.
5. Затем сам запусти `npm run db:push -- --yes` и проверь реальным запросом, что таблица создана.

Не проси личный access token для обычной локальной работы. Он даёт права Supabase-аккаунта на
все доступные ему проекты. Никогда не используй один общий token для детей и не сохраняй
`SUPABASE_ACCESS_TOKEN` или `SUPABASE_DB_PASSWORD` в репозитории, `.env.local` или чате.

## Чтобы проект не «уснул» (keep-alive)

Бесплатный Supabase **ставит проект на паузу после недели без запросов** — и живая ссылка
перестаёт работать, когда автор перестаёт заходить. Чтобы этого не было, в репозитории есть
`.github/workflows/keep-awake.yml` — он раз в день делает реальный select к таблице `entries`.

- **Не удаляй** этот файл и его расписание.
- Значения из `.env.local` НЕ коммить в YAML. Сохрани их в GitHub Actions Secrets:
  `VITE_SUPABASE_URL` → `SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (в нём может быть новый
  publishable key или legacy anon key) → `SUPABASE_ANON_KEY`.
- Делай это через `gh secret set`, не печатай значения в терминал или ответ пользователю.
- **Никогда** не используй `service_role` ключ.
- Перед завершением: сделай такой же запрос локально, затем push, `workflow_dispatch` и проверь
  зелёный run с `HTTP 200`. Если `gh` не авторизован — остановись и попроси ментора сделать
  `gh auth login`. Не говори ребёнку, что всё готово, пока удалённый run не прошёл.
- Для всей настройки уже есть команда `npm run keep-awake:setup`. Запусти её после `db:push`.
  Она не печатает ключи, сама ставит GitHub Secrets, запускает workflow и проверяет его лог.
  Если команда упала — исправь причину и запускай снова до успешной ссылки на run.

## Как добавить таблицу или поле (важно — делай ТОЛЬКО так)

```bash
npm run db:new -- add_scores      # создаст supabase/migrations/<timestamp>_add_scores.sql
```
Впиши SQL в новый файл (создание таблицы + `enable row level security` + политики «только свои строки»),
затем:
```bash
npm run db:push                   # применит к облачной базе
```
Дай пользователю текст SQL, который ты записал, и скажи запустить `npm run db:push`. НЕ предлагай
открывать SQL Editor вручную.


## AI внутри продукта — используем Gemini (бесплатный ключ)

В проекте уже есть готовая Edge Function `supabase/functions/ai` — она вызывает **Google Gemini**
(бесплатный ключ). НЕ создавай новую AI-функцию и НЕ вызывай модель напрямую с фронта.

- Ключ берётся из секрета `GEMINI_API_KEY` (на сервере, не в коде). Бесплатный ключ:
  https://aistudio.google.com/apikey
- Запуск: `npm run ai:secret -- GEMINI_API_KEY=...` затем `npm run ai:deploy`.
- Вызов с фронта:
  ```ts
  const { data } = await supabase.functions.invoke('ai', {
    body: { prompt: 'текст запроса', system: 'роль и правила (system prompt)' },
  });
  // data.text — ответ модели
  ```
- Когда добавляешь AI-фичу: пиши логику во фронте + используй эту функцию. Если нужна другая
  модель — поменяй `MODEL` в `supabase/functions/ai/index.ts`, не плоди новые функции.

## Подсказки под частые задачи

- **AI внутри продукта**: только через готовую функцию `ai` (Gemini), ключ на сервере.
- **Файлы/картинки от пользователя**: используй Supabase Storage, не храни бинарь в таблицах.
- **Игры**: игровую логику (canvas, физика) держи отдельно от UI. При смене дизайна не ломай игровой процесс.
- **Дизайн**: спокойный, один акцентный цвет, много воздуха, шрифт Inter. Без неона.
