# 🤖 Промпты для Codex

Codex меняет **настоящий код** в твоём проекте. Чтобы он делал это правильно — дай ему контекст
один раз, а потом проси фичи по шаблонам. Всё в `[квадратных скобках]` — заменяй на своё.

---

> 💡 В проекте есть файл **AGENTS.md** — Codex читает его сам и уже знает стек, правила и команды.
> Контекст-промпт ниже — короткое напоминание на случай, если ассистент его не подхватил.

## 0️⃣ Контекст-промпт (вставь ПЕРВЫМ, один раз в начале проекта)

```
Это мой проект на стеке: Vite + React + TypeScript + Supabase + Vercel.
Следуй правилам из AGENTS.md. Коротко:
- Supabase уже подключён в src/lib/supabase.ts (ключи в .env.local). Используй ЕГО.
- Базу меняй ТОЛЬКО миграциями: npm run db:new -- имя → впиши SQL → сам сделай dry-run,
  проверь RLS → сам примени db:push и проверь результат.
  Никакого ручного SQL Editor. На каждую таблицу — RLS «только свои строки».
- Код на TypeScript, без any. Маленькие компоненты. Перед концом — npm run build.
- Не трогай ключи и .env, не коммить .env.local.
- Объясняй кратко, что поменял и в каких файлах.
Подтверди, что понял, и жди мою первую задачу.
```

---

## 🔁 Главное правило цикла

После КАЖДОГО промпта Codex: **посмотри что изменилось → запусти `npm run dev` → работает? →
`git push`**. Сломалось — попроси починить, не коммить сломанное.

## 🗓️ Промпты по дням (шаблоны)

### День 2 — подключи Supabase полностью
```
Подключи этот проект к моему Supabase полностью сам. Я не должен вводить команды или искать ID.

1. Проверь, что .env.local содержит VITE_SUPABASE_URL и VITE_SUPABASE_ANON_KEY. Не печатай их.
2. Запусти npm run db:login. Остановись только чтобы я подтвердил вход в браузере.
3. Сам извлеки project ref из VITE_SUPABASE_URL и запусти
   npm run db:link -- --project-ref REF.
4. Если терминал запросит пароль базы, попроси меня ввести его прямо в терминал. Не проси
   отправлять пароль или access token в чат и не сохраняй их в проекте.
5. Запусти npm run db:push -- --dry-run. Проверь ожидаемые миграции, RLS и политики доступа.
6. Если проверка безопасна, сам запусти npm run db:push -- --yes.
7. Проверь реальным запросом, что таблица entries существует. Если что-то не работает — сам
   найди причину и исправляй до успешной проверки.

Не создавай и не проси SUPABASE_ACCESS_TOKEN для обычной локальной работы. Никогда не используй
общий token между детьми. В конце кратко сообщи, что login, link, migrations и проверка успешны.
```

## 😴 Keep-alive без отладки ребёнком — вставь один раз в День 2

> Отправь этот промпт **после `npm run db:push`**, когда `.env.local` уже заполнен и таблица
> `entries` создана. Codex должен сам настроить, протестировать и починить всё до зелёного run.

```
Настрой keep-alive полностью сам. Ребёнок не должен ничего дебажить.

1. Убедись, что таблица `entries` создана через `npm run db:push`.
2. Запусти `npm run keep-awake:setup`. Команда сама безопасно прочитает `.env.local`, проверит
   что ключ не `service_role`, сделает локальный ping, сохранит GitHub Actions Secrets через
   stdin, запустит удалённый workflow и проверит его лог.
3. Если команда говорит, что `gh` не авторизован — остановись и одной строкой попроси ментора
   выполнить `gh auth login`. После авторизации снова запусти команду.
4. Если команда упала по другой причине — сам найди и исправь причину. Если менял код или
   workflow: `npm run build` → commit → push. Затем снова `npm run keep-awake:setup`.
5. Повторяй до строки `Keep-alive verified: <ссылка>`. В конце сообщи только: что ежедневное
   расписание активно и ссылку на успешный run. Значения из `.env.local` не показывай.
```
